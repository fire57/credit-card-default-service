# План A/B-тестирования

## Цель

Сравнить две версии модели прогнозирования дефолта:

- `v1` - контрольная модель, Logistic Regression;
- `v2` - тестовая модель, Random Forest.

Цель теста - проверить, улучшает ли `v2` качество выявления будущих дефолтов без недопустимого роста ложных отказов надежным клиентам.

## Разделение трафика

Базовая схема:

- 50% запросов - контрольная группа `v1`;
- 50% запросов - тестовая группа `v2`.

В API это реализовано двумя способами:

- явный выбор через `model_version`;
- автоматическое распределение, если `model_version` не передан.

Если передан `ab_key`, распределение стабильно: один и тот же клиент остается в одной группе на протяжении теста.

## Продолжительность

Рекомендуемая длительность: 2-4 недели.

Причины:

- нужно накопить достаточно заявок и дефолтных исходов;
- недельные циклы помогают учесть поведение клиентов в будни и выходные;
- слишком короткий тест может дать случайный результат.

В реальном кредитном скоринге дефолт может проявляться с задержкой, поэтому для финальной бизнес-оценки нужен последующий период наблюдения.

## Метрики

### Основная метрика

`F1-score` для класса дефолта.

Почему она подходит:

- класс дефолта обычно миноритарный;
- важен баланс между `Precision` и `Recall`;
- простая accuracy может быть обманчивой.

### Дополнительные метрики

`Recall` для класса дефолта:

- показывает, какую долю реальных дефолтов модель смогла найти;
- важен для снижения кредитных потерь.

`Precision` для класса дефолта:

- показывает, насколько часто срабатывания модели действительно являются дефолтами;
- важен, чтобы не отказывать слишком многим надежным клиентам.

### Бизнес-метрики

`Expected Loss`:

```text
Expected Loss = PD * EAD * LGD
```

`Approval Rate` при фиксированном уровне риска:

```text
Approval Rate = approved_clients / all_clients
```

Успешная модель должна снижать expected loss или увеличивать approval rate без роста фактических дефолтов.

## Статистический анализ

Для долевых метрик, например доли дефолтов среди одобренных заявок или approval rate, можно использовать z-test для двух пропорций.

Для средних финансовых метрик, например expected loss на клиента, можно использовать t-test или bootstrap, если распределение сильно скошено.

Доверительные интервалы:

- для долей - нормальная аппроксимация или Wilson interval;
- для средних финансовых эффектов - bootstrap 95% CI.

## Критерий успешности

Модель `v2` считается успешной, если:

- `F1-score` для класса дефолта статистически значимо выше, чем у `v1`;
- `Recall` не ниже контрольной модели или выше при приемлемом `Precision`;
- expected loss не увеличивается;
- нет деградации технических метрик API: latency, error rate, стабильность ответов.

## Влияние на дизайн сервиса

A/B-тест требует, чтобы сервис:

- поддерживал несколько версий модели одновременно;
- логировал `model_version` и `ab_group`;
- сохранял стабильное распределение клиентов по группам;
- возвращал единый формат ответа независимо от версии модели.

Именно поэтому `/predict` возвращает `model_version`, `ab_group` и `selection_method`.
