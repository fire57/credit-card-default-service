# Архитектура сервиса

## Выбранный подход

Для учебного проекта выбран монолитный ML-сервис:

- Flask API;
- загрузка моделей из локальной папки `models/`;
- синхронный инференс в эндпоинте `/predict`;
- JSON-логирование в stdout.

Такой подход подходит для текущего масштаба: одна команда, один датасет, две версии модели и простая интеграция через HTTP. Монолит проще развернуть, тестировать и объяснить в рамках production-like проекта.

## Монолит vs микросервисы

Монолит выбран потому что:

- у сервиса одна основная бизнес-функция - прогноз дефолта;
- нет отдельной команды для каждого компонента;
- модель небольшая и быстро выполняет инференс;
- проще обеспечить воспроизводимость и локальный запуск.

Микросервисный подход стал бы полезен при росте системы:

- отдельный сервис признаков;
- отдельный сервис инференса;
- отдельный сервис логирования и мониторинга;
- отдельный batch-сервис для массового скоринга;
- отдельный model registry.

В промышленной банковской среде микросервисы помогают независимо масштабировать API, обработку событий, хранение фичей и мониторинг.

## A/B-тестирование в архитектуре

API поддерживает две версии модели:

- `v1` - контрольная модель;
- `v2` - тестовая модель.

Если клиент передает `model_version`, сервис использует указанную модель. Если версия не передана, сервис назначает запрос в `v1` или `v2` по схеме 50/50. При наличии `ab_key` распределение стабильно: один и тот же клиент попадает в одну и ту же группу.

Это позволяет проверять новую модель без отдельного сервиса и без изменения клиентского API.

## RabbitMQ и очереди сообщений

RabbitMQ мог бы пригодиться при масштабировании:

- асинхронная обработка batch-предсказаний;
- отправка событий инференса в отдельный сервис логирования;
- постановка тяжелых задач скоринга в очередь;
- повторная обработка запросов при временных сбоях downstream-систем;
- интеграция с антифродом, CRM или хранилищем решений.

Пример сценария:

1. API принимает заявку клиента.
2. Быстро возвращает `request_id`.
3. Событие попадает в RabbitMQ.
4. Worker забирает событие, делает скоринг и пишет результат в базу.
5. Внешняя система забирает результат по `request_id`.

В текущем проекте RabbitMQ описан концептуально, потому что синхронный инференс быстрее и проще для учебной демонстрации.

## Логирование и мониторинг

Сервис пишет JSON-логи в stdout. Для каждого успешного прогноза логируются:

- `event`;
- `request_id`;
- `model_version`;
- `ab_group`;
- `prediction`;
- `probability`;
- `duration_ms`.

В production эти логи можно собирать через:

- Docker logging driver;
- Filebeat / Logstash;
- ELK-стек;
- Loki + Grafana;
- облачный мониторинг.

На основе логов можно отслеживать:

- задержку API;
- долю ошибок;
- распределение вероятностей дефолта;
- долю трафика между `v1` и `v2`;
- drift во входных признаках.

## uWSGI + NGINX

В production Flask обычно не запускают встроенным dev-сервером.

Типовая схема:

- NGINX принимает HTTP-трафик, обрабатывает TLS, лимиты и keep-alive;
- uWSGI или Gunicorn запускает Python-приложение несколькими worker-процессами;
- Flask отвечает только за бизнес-логику API.

В этом проекте Dockerfile использует Gunicorn как production-like WSGI-сервер. NGINX не обязателен для учебной задачи, но может быть добавлен как reverse proxy.

## ONNX-ML

Модель sklearn можно преобразовать в ONNX-ML для ускорения и стандартизации инференса.

Возможный путь:

1. Установить `skl2onnx` и `onnxruntime`.
2. Загрузить обученный sklearn pipeline.
3. Описать типы входных признаков.
4. Вызвать `convert_sklearn`.
5. Сохранить `.onnx`.
6. В API заменить `joblib.load` на `onnxruntime.InferenceSession`.

ONNX полезен, когда модель нужно запускать вне Python-среды, уменьшить зависимость от sklearn или унифицировать инференс между разными платформами.

## DVC

DVC помогает версионировать данные и артефакты моделей:

- хранить большие CSV и модели вне Git;
- связывать версию данных с версией кода;
- воспроизводить обучение по `dvc repro`;
- хранить артефакты в S3, GCS, MinIO или другом remote storage.

Для этого проекта DVC мог бы контролировать:

- `data/raw/UCI_Credit_Card.csv`;
- `models/model_v1.joblib`;
- `models/model_v2.joblib`;
- `models/metrics.json`.

## MLflow

MLflow нужен для отслеживания экспериментов:

- параметры моделей;
- метрики;
- версии артефактов;
- сравнение запусков;
- model registry;
- stage-механика: `Staging`, `Production`, `Archived`.

В данном проекте MLflow мог бы заменить ручной `metrics.json` и упростить выбор модели-кандидата для A/B-теста.

## Бизнес-метрики

Технические метрики важны, но заказчику нужны финансовые эффекты.

### Ожидаемые финансовые потери

Можно оценить:

```text
Expected Loss = PD * EAD * LGD
```

Где:

- `PD` - вероятность дефолта из модели;
- `EAD` - сумма под риском, например кредитный лимит;
- `LGD` - доля потерь при дефолте.

Модель успешна, если помогает снижать суммарный expected loss при приемлемом уровне одобрений.

### Доля одобренных заявок при фиксированном риске

Можно выбрать порог вероятности дефолта и сравнить:

```text
Approval Rate = approved_clients / all_clients
```

Бизнес-цель: увеличить долю одобрений при сохранении заданного уровня дефолтов или expected loss.
